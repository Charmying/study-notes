# Redis

在現代應用程式開發中，「速度」與「即時性」往往是影響使用者體驗的關鍵因素。當應用程式需要處理大量資料、快速回應查詢，或支援高併發操作時，傳統關聯式資料庫有時無法滿足需求。在這樣的背景下，Redis 作為以記憶體為主的高效能資料庫，受到廣泛使用。

<br />

## Redis 簡介

Redis (Remote Dictionary Server) 是一個由 Salvatore Sanfilippo 於 2009 年開發的開源記憶體資料結構儲存系統，同時支援資料庫、快取 (Cache) 以及訊息代理 (Message Broker) 的功能。Redis 主要將資料保存在記憶體中，因此讀取速度極快，遠快於傳統硬碟存取的資料庫。此外，Redis 也支援將資料持久化到硬碟，以避免資料遺失。

Redis 支援多種資料結構，例如

- String (字串)：最基本的資料類型，可儲存文字或二進位資料。

- Hash (雜湊)：鍵值對集合，適合儲存物件型資料。

- List (列表)：有序序列，可進行推入、彈出操作。

- Set (集合)：無序、不重複元素集合。

- Sorted Set (有序集合)：每個元素有分數，可依分數排序。

- Bitmap (點陣圖)：適合統計大規模使用者是否發生某事件。

- HyperLogLog (HLL)：適合估算獨立訪客數量，節省記憶體。

- Stream (串流)：適合事件驅動架構或訊息串流處理。

| 資料型別 | 用途 | 特點 |
| - | - | - |
| String | 文字/二進位資料 | 最基本型別，操作簡單 |
| Hash | 物件型資料 | 鍵值對集合，適合存儲物件 |
| List | 序列資料 | 支援推入/彈出操作 |
| Set | 唯一集合 | 無序、不重複元素 |
| Sorted Set | 排行榜 | 可依分數排序 |
| Bitmap | 統計事件 | 高效記憶體使用 |
| HyperLogLog | 訪客統計 | 估算獨立元素，節省記憶體 |
| Stream | 事件串流 | 適合事件驅動或訊息串流 |

## Redis 的優缺點

### 優點

- 高效能：資料主要儲存在記憶體中，讀寫速度可達微秒級，適合高頻率存取場景。

- 靈活的資料結構：多樣化資料型別，方便建構複雜應用，例如：排行榜、即時統計、訊息隊列等。

- 簡單易用：提供簡潔明瞭的指令介面，支援多種程式語言客戶端，例如：Java、Python、Node.js、Go 等。

- 持久化：支援快照 (RDB) 與附加檔案 (AOF) 兩種持久化方式，可依需求選擇

    - RDB：生成快照，適合定期備份及快速重啟，但可能丟失最近的修改。

    - AOF：即時記錄每個操作，資料更完整，但檔案較大且恢復稍慢。

    - 混合模式 (RDB + AOF)：兼顧速度與資料安全，Redis 7 後對 AOF 重寫效率有改進，實務中常使用。

- 高可用與分布式

    - Redis Sentinel：提供監控與自動故障轉移功能，確保高可用。

    - Redis Cluster

        - 功能：支援資料分片與水平擴展；每個分片可配置 Master-Slave 節點，提高可靠性與容錯能力。

        - 一致性：採用最終一致性 (Eventual Consistency) 策略，開發者需注意資料同步延遲與容錯設計。

- 支援原子操作：許多操作是原子性的，保證資料一致性，降低開發複雜度。

### 缺點

- 記憶體消耗大：資料主要存於記憶體，資料量大時需要大量 RAM，成本較高。

- 資料持久化延遲：極端情況下，可能有短暫遺失資料的風險。

- 單線程限制：Redis 核心操作是單線程，雖然透過事件循環與非阻塞 I/O 提高效能，但單個實例 CPU 利用率受限，需透過分片或多實例擴展。

- 資料模型簡單：不支援複雜查詢與關聯操作，需要開發人員在應用層實現。

- 安全性考量：部署需注意防火牆、認證 (AUTH)、ACL 控制，避免未授權存取。

## Redis 的應用場景

- 快取 (Cache)：快取資料庫查詢結果或計算結果，減少對主資料庫的壓力，提高系統響應速度。

- 會話管理 (Session Store)：用於存儲使用者登入狀態、Token 或 Session 資料。

- 計數與統計：使用 Hash、Sorted Set 或 HyperLogLog，進行即時統計、計數器、熱點追蹤等操作。

- 訊息隊列與事件流：透過 List 或 Stream 實現任務隊列、事件串流或即時訊息傳遞。

- 排行榜與社群功能：利用 Sorted Set 實現遊戲排行榜、人氣排行或追蹤用戶行為。

- 即時分析與監控：快速聚合資料，提供即時監控與分析結果。

<br />

## Redis 實務建議

- 在高併發系統中，建議將熱點資料快取到 Redis，並配合過期策略與監控，避免記憶體爆滿。

- 結合 Redis Cluster 或多實例部署，可支援大資料量與高可用需求。

- 注意資料持久化策略，根據業務需求選擇 RDB、AOF 或混合模式，平衡速度與資料安全。

- 強化安全性配置，設定防火牆、AUTH 與 ACL，降低潛在攻擊風險。

<br />

## 總結

Redis 作為記憶體型資料庫，以高速、高效特性成為現代網路應用程式不可或缺的工具。無論作為快取、訊息代理、會話管理，還是即時資料分析，Redis 都能顯著提升系統性能。然而，在使用 Redis 時，仍需注意記憶體資源管理、資料持久化策略及安全性配置，才能在高效能與穩定性之間取得最佳平衡。
