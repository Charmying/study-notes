# 憑證綁定 (Certificate Pinning)

在現今的網路環境中，HTTPS 已是網路安全的基本標準，但僅憑 TLS/SSL 憑證仍無法完全避免攻擊者透過偽造憑證或中間人攻擊。為了進一步強化安全性，憑證綁定 (Certificate Pinning) 技術便應運而生。

憑證綁定指的是在應用程式 (App 或瀏覽器) 中，事先內建或指定可信任的伺服器憑證或公鑰，當連線建立時，除了正常的 TLS 驗證外，還會額外檢查伺服器回應的憑證是否與「內建的憑證」或「對應的公鑰」相符。

若不相符，連線即會被拒絕。這樣做可以避免以下風險

- 攻擊者取得由不受信任或被濫用的憑證機構 (CA) 簽發的假憑證。

- 避免使用者裝置被植入惡意的根憑證，導致連線遭竊聽。

簡單來說，就是「除了信任 CA，還要確認憑證是否與預期一致」。

<br />

## 憑證綁定的運作原理

一般的 TLS 連線驗證流程

1. 客戶端 (App/瀏覽器) 向伺服器發出連線請求。

2. 伺服器回傳憑證。

3. 客戶端驗證憑證是否合法 (CA 簽發、未過期、鏈結完整)。

若採用憑證綁定，則在傳統 TLS 驗證之外，會額外加入比對步驟

4. 客戶端將伺服器憑證或公鑰與「內建的憑證 (或雜湊值)」比對，若不相符，即便憑證是合法 CA 簽發，也會中斷連線。

<br />

## 綁定的種類

綁定技術主要有兩種做法

- 憑證綁定 (Certificate Pinning)

    - 直接在應用程式內建完整的伺服器憑證。

    - 驗證時比對憑證內容是否一致。

    - 缺點：只要伺服器憑證更新 (例如：快到期換新)，就必須更新應用程式，這在 App 特別麻煩，因為必須重新送審與上架。

- 公鑰綁定 (Public Key Pinning)

    - 只比對憑證的公鑰 (或其雜湊值)。

    - 即使伺服器更新憑證，只要使用相同的公鑰 (CSR 產生時不變)，驗證仍會通過。

    - 相對較有彈性，也更常被使用。

<br />

## 憑證綁定的常見應用場景

- 行動應用程式 (App)

    在金融、支付、醫療等高安全需求的 App 中，常會實作憑證綁定，避免用戶在公共 Wi-Fi 或被安裝惡意根憑證時遭攔截。

- API 安全性

    伺服器端 API 提供者可要求使用者端 App 實作 Pinning，確保 API 呼叫一定是直連官方伺服器，而不是被轉導，這對提供行動 App 或第三方服務的 API 特別重要，可防止流量被惡意代理伺服器攔截。

- 防範 MITM 攻擊

    攻擊者可能利用被濫用的 CA 簽發假憑證，或利用 Proxy 工具攔截 HTTPS 流量。Pinning 可有效降低這類風險。

<br />

## 憑證綁定的實作方式

- iOS

    - 使用 `URLSessionDelegate` 的 `didReceive challenge` 方法進行憑證或公鑰比對。

- Android

    - 使用 `Network Security Config` (XML 設定檔) 或在程式碼中透過 `SSLSocketFactory` 驗證憑證。

- Web (瀏覽器端)

    - 曾經有 HTTP Public Key Pinning (HPKP) 技術，但是因為管理風險過高 (若設定錯誤會導致整個網站無法訪問)，目前已被主流瀏覽器棄用。

    - 現在更建議使用 Expect-CT 或 Certificate Transparency (憑證透明機制) 來達成類似效果，但 Expect-CT 也已經被 Chrome 宣布棄用 (2021 年起)，所以現行主要依靠 Certificate Transparency。

<br />

## 憑證綁定的優缺點

### 優點

- 提升安全性，降低憑證被濫用的風險。

- 防止中間人攻擊與偽造憑證。

### 缺點

- 維護成本高：憑證更新需要重新發佈 App (若採用憑證綁定 而非公鑰綁定)。

- 容易造成誤鎖：若 Pinning 設定錯誤，合法的連線也可能被拒絕。

- 無法完全取代 CA：Pinning 並非要取代 PKI，而是作為額外的一層防護機制。

<br />

## 憑證綁定的最佳實務建議

- 建議使用公鑰綁定，降低因憑證更新造成的維護問題。

- 設定多組 Pin (Fallback Key)：避免單一金鑰失效導致系統全面中斷，也建議預留至少一組未來金鑰 (Backup Key)，以便在憑證異動時立即切換。

- 搭配監控機制：若 Pinning 驗證失敗，應提供適當錯誤處理或上報機制。

- 測試與演練：在正式發佈前，先模擬憑證更新與錯誤情境，避免因設定錯誤造成服務中斷，特別是要測試在憑證過期、憑證撤銷或更換金鑰的情境下，應用程式是否能正常處理。

<br />

## 總結

憑證綁定是一種強化 TLS 安全的機制，透過在客戶端內建憑證或公鑰來降低被偽造憑證攻擊的風險。

憑證綁定對金融、支付、醫療等高敏感應用特別重要。但 Pinning 若管理不慎，反而可能造成連線失敗與維護困難。

總而言之，憑證綁定能顯著提升傳輸安全，但實務上需權衡安全與維運成本。因此，實務上更建議採用公鑰綁定，並搭配金鑰輪替策略與監控，才更能達到安全且穩健的效果。
