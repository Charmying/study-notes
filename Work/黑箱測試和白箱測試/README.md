# 黑箱測試和白箱測試

在軟體測試中，黑箱測試 (Black Box Testing) 和白箱測試 (White Box Testing) 是兩種常見的方法，分別側重於不同的測試視角和技術細節。

<br />

## 黑箱測試 (Black Box Testing)

### 定義

黑箱測試是一種測試方法，測試人員在不需要了解程式內部運作的情況下，根據系統的規格說明書來驗證軟體的功能是否正確。這種方法把軟體看作一個「黑箱」，不考慮內部結構和程式碼，僅專注於輸入和輸出。

### 特點

- 不需要了解程式內部結構：測試者可以是外部人員，不需要具備程式設計技能。

- 基於需求和功能：根據需求規格書來設計測試案例。

- 重點放在軟體的功能：確保所有功能能夠如預期運作。

### 優點

- 模擬真實用戶的操作：能夠更好從使用者角度出發，發現功能上的問題。

- 適合功能驗證：能有效檢測功能是否滿足需求。

### 缺點

- 覆蓋率有限：由於無法檢查程式碼，可能無法覆蓋所有執行路徑。

- 缺乏細節：無法洞察內部流程上的錯誤或性能問題。

### 常見的黑箱測試技術

- 等價類劃分 (Equivalence Partitioning)：將輸入數據劃分為若干等價類別，並選擇每個類別中的代表值進行測試，以減少測試用例數量。

- 邊界值分析 (Boundary Value Analysis)：專注於測試邊界值 (例如：最小值、最大值、邊界之內的值等)，這是最容易出錯的地方。

- 決策表測試 (Decision Table Testing)：使用決策表來設計測試案例，這樣可以確保覆蓋所有可能的條件組合和結果。

- 狀態遷移測試 (State Transition Testing)：根據系統的狀態轉換圖進行測試，確保系統在不同狀態之間轉換時能正常工作。

- 聯合測試 (Integration Testing)：測試不同模組或系統之間的交互，確保能夠正確協同工作。

- 隨機測試 (Fuzz Testing)：通過生成隨機的輸入數據來測試系統，目的在找出不正確處理異常數據的情況。

<br />

## 白箱測試 (White Box Testing)

### 定義

白箱測試是一種測試方法，又叫做透明箱測試，測試人員需要了解程式的內部結構和處理流程，並根據程式碼進行測試。這種方法將程式視為一個「透明的白箱」，允許檢查內部運作。

### 特點

- 需要了解程式內部結構：測試者通常是開發人員或具有程式設計背景的測試人員。

- 基於程式碼和處理流程：測試用例的設計是基於源程式碼。

- 重點放在程式的內部處理流程：檢查所有可能的路徑和條件。

### 優點

- 高覆蓋率：可以覆蓋到程式碼的每一行，找到隱藏的錯誤。

- 深入檢查內部處理流程：能夠發現程式處理流程和結構上的問題。

### 缺點

- 成本較高：需要詳細的程式知識，測試準備和執行的時間較長。

- 難以模擬用戶行為：過於專注於內部細節，可能忽略使用者體驗。

### 常見的白箱測試技術

- 語句覆蓋 (Statement Coverage)：確保測試用例能執行每一行程式碼至少一次，檢查每行程式碼的正確性。

- 分支覆蓋 (Branch Coverage)：測試所有的條件分支，確保每個分支條件都能測試到，這樣有助於抓到流程錯誤。

- 路徑覆蓋 (Path Coverage)：測試所有可能的路徑，包括循環和條件的不同組合，確保所有處理路徑都被檢查過。

- 錯誤推導測試 (Error Guessing)：基於經驗和常見的錯誤模式進行測試，來發現系統中可能存在的缺陷。

- 單元測試 (Unit Testing)：針對最小的程式碼單元進行測試，確保每個單元都能正常運作。

6. 程式碼覆蓋測試 (Code Coverage Testing)：使用工具來測量測試過程中程式碼的覆蓋程度，幫助確保測試的全面性。

## 總結

- 黑箱測試適合測試系統的功能性需求，主要從用戶角度驗證系統是否滿足預期需求，而且不需要了解系統內部結構。

- 白箱測試側重測試系統內部處理流程和程式碼，主要針對開發人員或測試人員來說明系統內部的運作是否如預期並能檢測到更多潛在的錯誤。

兩者相輔相成，能夠全面提升軟體的質量和可靠性。通常在開發過程中，這兩種測試方法會結合使用，確保產品的功能性和內部質量都達到預期標準。
