# Jaeger

在現代微服務架構中，應用程式通常由多個服務組成，每個服務可能部署在不同的伺服器或容器中。雖然微服務提升了系統的彈性與可維護性，但也帶來了一個挑戰：當系統出現效能問題或錯誤時，很難快速追蹤問題的來源。這時，分散式追蹤 (Distributed Tracing) 就成為了不可或缺的工具，而 Jaeger 正是其中一個廣泛使用的解決方案。

<br />

## Jaeger 簡介

Jaeger 是由 Uber 開源的一套分散式追蹤系統，旨在幫助開發人員監控和分析微服務架構的效能與行為。

Jaeger 能夠

- 追蹤請求流程：從使用者發起請求，到系統中各個微服務的處理過程，都能被完整記錄。

- 分析延遲瓶頸：找出哪些服務或哪個環節造成請求延遲。

- 視覺化呈現：透過圖形化介面，將請求流程與時間軸直觀呈現，方便排查問題。

- 分佈式監控：結合 metrics (指標) 與 logs (日誌)，提供完整的系統監控視角。

簡單來說，Jaeger 可以幫助「看到每一個微服務如何處理請求」，並找出效能瓶頸或錯誤點。

<br />

## Jaeger 的核心概念

使用 Jaeger 時，有幾個核心概念需要理解

- Trace (追蹤)

    Trace 是完整的一次請求流程。例如，使用者在網站點擊「下單」，整個流程可能涉及前端、API 伺服器、支付服務與資料庫。這整個請求就是一個 Trace。

- Span (時間段)

    Span 是 Trace 中的單一操作，例如：「呼叫支付服務 API」或「資料庫查詢」。每個 Span 都會記錄開始與結束時間，以及相關的標籤 (tags) 和日誌 (logs)。

- Parent-Child 關係

    多個 Span 之間可能有父子關係。例如：前端的 API 請求是父 Span，而呼叫資料庫查詢是子 Span。這樣可以清楚呈現請求的流程與層級。

- Context Propagation (上下文傳遞)

    為了在微服務之間保持追蹤資訊，每個請求都會攜帶 Trace Context，讓下游服務能夠延續這次 Trace。

## Jaeger 的架構

Jaeger 的系統架構主要分為以下幾個部分

- Client Libraries (客戶端程式庫)

    各種程式語言 (例如：Go、Java、Python、Node.js 等) 都有 Jaeger 的 SDK，負責收集 Span 並將資料送到 Collector。

- Agent

    佈署在每個主機上，接收應用程式發送的 Span，並將資料轉發給 Collector。Agent 通常以 UDP 傳輸，減少效能影響。

- Collector

    負責接收 Agent 或 SDK 傳送過來的資料，進行處理後存入後端資料庫。

- Storage Backend (儲存後端)

    Jaeger 支援多種儲存方式，例如：Elasticsearch、Cassandra 或 Kafka，用來保存 Trace 資料，便於查詢與分析。

- Query & UI (查詢與介面)

    提供 Web UI，開發人員可以透過 Jaeger 搜尋 Trace、分析請求流程、查看延遲來源，以及比對不同請求的效能。

<br />

## Jaeger 的功能與優勢

- 效能監控與分析：可以清楚看到每個服務的延遲與瓶頸，幫助改善效能。

- 錯誤追蹤：當系統發生錯誤時，能快速定位到哪個服務或哪個環節出問題。

- 可視化追蹤：透過 UI 呈現請求的時間線，讓問題一目了然。

- 整合性強：與 Prometheus、Grafana 等監控工具搭配，可形成完整的 Observability (可觀察性) 解決方案。

- 多語言支援：支援多種程式語言與框架，方便在異質微服務環境中使用。

<br />

## 總結

在微服務架構下，單靠傳統的日誌或監控指標，已經無法完整掌握系統的運作情況。Jaeger 作為分散式追蹤系統，提供了從請求追蹤到效能分析的完整方案，讓開發人員能夠快速定位問題、提升系統效能，並增強整體使用者體驗。

對於任何需要高可觀察性、效能可分析的微服務系統來說，Jaeger 都是一個強而有力的工具。
