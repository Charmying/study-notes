# MVP (最小可行產品)

MVP (Minimum Viable Product，最小可行產品) 在新創領域或產品開發領域中幾乎是每個人都會聽到的關鍵字。

核心概念是：用最小的成本，打造一個具備最基本功能的產品原型，並快速投入市場測試，以驗證想法是否可行。

<br />

## MVP 的核心精神

MVP 並不是一個「半成品」或「隨便做做的東西」，而是先聚焦於做一個能解決用戶核心需求、可以真實上線測試的最簡版本產品，再依照市場回饋逐步改進，而不是一開始就做一個「完美的大產品」。

透過 MVP，可以

- 快速驗證市場需求：避免花費大量時間與資源在一個沒人需要的產品上。

- 蒐集真實用戶回饋：幫助團隊優化產品方向。

- 降低開發成本與風險：先確認「這件事是否值得做」再全力投入。

<br />

## MVP 的三大特徵

- Minimum (最小)

    - 功能只做「最核心、最關鍵」的部分。

    - 不追求完整，只要能運作並解決主要問題即可。

- Viable (可行)

    - 必須真的能被使用者拿來操作。

    - 不是 Wireframe、不是 Mockup，而是能實際運行的產品。

- Product (產品)

    - 不是單純的原型圖，而是能在市場上測試的版本。

<br />

## MVP 的常見型態

在不同產業與階段，MVP 的形式可能大不相同，以下是幾種常見方法

- 手動服務 (Wizard of Oz)

    前端看起來像自動化系統，但背後其實是人工操作。

    例如：早期的線上訂餐服務，可能是網站接單後，創辦人親自打電話去餐廳訂餐。

- 登錄頁 (Landing Page)

    製作一個簡單的產品介紹頁面，讓有興趣的人先留下聯絡方式或預先註冊，以測試市場需求。

- 樣品或原型 (Prototype)

    透過低成本工具 (例如：Figma、Adobe XD、簡單互動 Demo) 驗證產品流程與使用體驗，屬於介於概念與 MVP 之間的測試方法。

- 管家式 MVP (Concierge MVP)

    由團隊手動提供產品的核心價值，模擬未來完整功能。

    例如：創辦人親自協助用戶完成任務，觀察是否有人願意為此付費。

<br />

## MVP 的實際案例

Uber (早期版)：最早版本只做「叫車」功能 → 沒有司機評價、沒有分攤費用、沒有外送。

Facebook (早期版)：一開始只開放給哈佛學生使用 → 僅有「建立個人檔案、加朋友」等基本功能，後續才逐步擴充。

Airbnb：創辦人先在自己的公寓放上簡單網站，測試是否有人願意付費住進陌生人家中，結果證明需求龐大。

Dropbox：在產品還沒完成前，創辦人用一支 Demo 影片展示核心功能，結果獲得大量等待名單。這嚴格來說是「假門測試 (Fake Door Test)」，而不是可操作的 MVP，但依舊實現了「用最小成本驗證需求」的目的。

<br />

## 總結

MVP 不是「做半套的產品」，而是一種快速驗證、降低風險、聚焦核心價值的產品開發策略。對於新創公司、產品經理或開發團隊而言，MVP 是一個必須理解並善用的工具。
