# Database Transaction

資料庫交易 (Database Transaction) 是一個資料庫管理系統 (DBMS) 中用來確保多個操作步驟能夠作為一個整體來執行的機制。這些操作要麼全部執行，要麼完全不執行 (完全不對資料庫產生影響)，確保資料庫在多個操作的過程中保持一致性和可靠性，而資料庫交易的核心特性可以用 ACID 原則來描述。

<br />

## 交易的基本概念

在現代資訊系統中，交易是一種用來處理和管理資料庫中複雜操作的工具。無論是金融交易、訂單處理或其他任何形式的數據操作，交易的主要目的是確保資料的一致性和可靠性，所以不管是系統故障、人為錯誤或是其他非預期問題，交易系統都能確保資料不會處於不一致或不完整的狀態。

<br />

## ACID 原則

- 原子性 (Atomicity)：

    原子性是指交易中的所有操作必須視為一個整體。也就是說，所有操作要麼全部完成，要麼全部失敗。在實際情況中，若交易中的任何一個操作失敗，整個交易返回原狀，並且資料庫將恢復到交易開始之前的狀態。例如：在銀行系統中，若在從一個帳戶扣款之後但在存入另一個帳戶之前系統崩潰，那麼整個交易應該返回原狀，確保沒有資金丟失。

- 一致性 (Consistency)：

    一致性是指交易必須將資料庫從一個一致狀態轉換到另一個一致狀態。也就是說，說交易不能違反資料庫的任何完整性約束。例如：若資料庫設有一個限制，使帳戶餘額不能為負，那麼任何交易都不能導致負餘額的情況出現。

- 隔離性 (Isolation)：

    隔離性確保交易的執行結果不會受到其他並發交易的影響。這在多使用者環境的中特別重要，因為通常有許多交易同時進行。隔離性允許交易在被其他交易影響之前完成，這可以通過使用不同的隔離級別來控制，例如：讀取已提交數據 (Read Committed) 或可重複讀取 (Repeatable Read)。

- 持久性 (Durability)：

    持久性保證一旦交易提交，其對資料庫的更改將永久保存。即使是系統崩潰或失去電力等狀況，已提交的交易數據應該可以從非揮發性儲存中恢復。這通常通過寫入日誌文件 (Log File) 來實現，以便在需要時進行恢復。

## 交易操作的範例

使用 SQL 和 Python 做一個交易操作的簡單範例

### SQL

```sql
-- 開始交易
START TRANSACTION;

-- 從帳戶A扣除100元
UPDATE accounts
SET balance = balance - 100
WHERE account_id = 'A';

-- 向帳戶B增加100元
UPDATE accounts
SET balance = balance + 100
WHERE account_id = 'B';

-- 檢查帳戶A是否有足夠的餘額
SELECT balance FROM accounts WHERE account_id = 'A';

-- 確保操作成功並提交交易
-- 若帳戶A的餘額足夠
IF (SELECT balance FROM accounts WHERE account_id = 'A') >= 0 THEN
    COMMIT;
ELSE
	-- 若帳戶A餘額不足則返回交易
	ROLLBACK;
END IF;
```

1. 開始交易：

    - 使用 `START TRANSACTION;` 開始一個新的交易。

2. 操作步驟：

    - 第一個操作是從帳戶 A 的餘額中扣除 100 元。

    - 第二個操作是向帳戶 B 的餘額中增加 100 元。

3. 驗證與提交：

    - 查詢帳戶 A 的餘額以確保扣款後餘額不為負數。

    - 若帳戶 A 的餘額足夠，則使用 `COMMIT;` 提交交易，將所有更改永久保存。

    - 若帳戶 A 的餘額不足，則使用 `ROLLBACK;` 返回交易，撤銷所有更改。

### Python & SQLite

```python
import sqlite3

# 連接到資料庫
connect = sqlite3.connect('bank.db')

try:
    # 開始交易
    connect.execute('BEGIN;')

	# 從帳戶A扣除100元
	connect.execute('UPDATE accounts SET balance = balance - 100 WHERE account_id = "A";')

	# 向帳戶B增加100元
	connect.execute('UPDATE accounts SET balance = balance + 100 WHERE account_id = "B";')

	# 確認帳戶A的餘額是否充足
	cursor = connect.execute('SELECT balance FROM accounts WHERE account_id = "A";')
	balance = cursor.fetchone()[0]

	if balance >= 0:
		# 提交交易
		connect.commit()
		print("交易成功！")
	else:
		# 若餘額不足，返回交易
		connect.rollback()
		print("交易失敗，餘額不足。")

except Exception as e:
	# 若有錯誤，返回交易
	connect.rollback()
	print("交易失敗，發生錯誤：", e)

finally:
	# 關閉資料庫連接
	connect.close()
```

1. 連接資料庫：使用 `sqlite3.connect()` 方法建立資料庫連接。

2. 開始交易：使用 `connect.execute('BEGIN;')` 開始交易。

3. 執行交易操作：執行一系列 `UPDATE` 操作來扣除和增加金額。

4. 檢查餘額：確認扣款後帳戶 A 的餘額是否充足。

5. 提交或返回交易：根據餘額情況使用 `connect.commit()` 或 `connect.rollback()` 來提交或返回交易。

6. 錯誤處理：使用 try-except 結構來處理可能發生的錯誤，確保即使發生例外情況也能返回交易。

7. 關閉連接：最後使用 `connect.close()` 關閉資料庫連接。

## 交易的運作機制

- 活躍狀態 (Active)：交易開始並且處於活躍狀態，此時交易正在進行中，所有操作尚未完成。

- 部分提交狀態 (Partially Committed)：當交易的最後一個操作完成後，交易進入部分提交狀態，此時交易等待提交。

- 失敗狀態 (Failed)：若交易在任何時候失敗，系統將會進入失敗狀態，並且開始回到所有已完成的操作。

- 中止狀態 (Aborted)：當交易因失敗而被回朔時，交易進入中止狀態，所有變更被撤銷，系統恢復到交易開始前的狀態。

- 提交狀態 (Committed)：當交易成功完成且所有變更被確認永久寫入資料庫時，交易進入提交狀態，此時變更無法撤銷。

## 交易在不同系統中的應用

- 銀行系統：

    在銀行系統中，交易用於處理涉及多個帳戶的複雜操作。例如：當客戶進行轉帳時，交易確保資金從一個帳戶正確轉移到另一個帳戶。若在過程中任何一步失敗，整個交易會被返回原狀，保證資金的安全性和完整性。

- 電子商務系統：

    在電子商務平台中，交易用於管理訂單處理、庫存管理和付款流程。當客戶下訂單時，交易確保庫存數量正確更新，付款已確認，以及訂單狀態正確變更。這些操作必須作為一個完整的交易進行，以出現庫存不足或重複付款等情況。

- 醫療資訊系統：

    在醫療資訊系統中，交易用於確保病患記錄的準確性和一致性。當醫生更新病患的病歷或開具處方時，交易確保所有資料同步更新，防止資料遺失或不一致。

## 隔離級別的選擇

在交易的隔離性中，不同的隔離級別提供不同程度的保護

- 讀未提交 (Read Uncommitted)：在這個隔離級別中，交易可以看到其他交易尚未提交的變更，可能導致髒讀 (Dirty Read) 問題。

- 讀已提交 (Read Committed)：在這個級別中，交易只能讀取其他交易已提交的數據，避免了髒讀，但可能會出現不可重複讀 (Non-repeatable Read) 問題。

- 可重複讀 (Repeatable Read)：該級別保證在交易期間，多次讀取相同數據將返回相同的結果，防止不可重複讀，但仍可能發生幻讀 (Phantom Read)。

- 序列化 (Serializable)：最高隔離級別，保證交易完全隔離，所有交易彷彿在串行執行，但這也降低了系統的並發性能。

## 交易管理的優缺點 

### 優點

- 資料完整性和一致性：交易通過 ACID 特性確保資料的一致性和完整性。即使在系統故障或並發操作中，也能防止資料不一致的情況發生。

- 原子性確保資料的安全性：交易的原子性保證所有操作要麼全部執行，要麼完全不執行，確保了資料庫不會處於部分更新的狀態，減少了資料損壞的風險。

- 故障恢復能力：交易管理系統通常包含故障恢復機制，能在系統崩潰或中斷後通過日誌文件恢復資料庫到一致狀態，確保資料持久性。

- 並發控制：隔離性特性允許多個交易並發執行，同時確保每個交易不會受到其他交易的影響。提高了系統的並發性能和使用效率。

- 簡化複雜操作管理：交易提供了一種簡化方式來處理複雜的資料操作，對開發人員來說可以更容易實現應用流程，減少程式碼錯誤的風險。

- ACID 模型提供穩定性：ACID 模型為企業提供了一個穩定的框架來處理資料，對於需要高可靠性的行業 (例如：金融或醫療等) 尤為重要。

### 缺點

- 性能開銷：實現 ACID 特性 (特別是高隔離級別) 時，交易管理會引入額外的性能開銷，降低系統的響應速度和整體性能。可能影響應用的擴展性和使用者體驗。

- 死鎖問題：在並發環境中，交易可能會導致死鎖，當多個交易互相等待資源時，整個系統可能會陷入停滯狀態，需要有效死鎖偵測和恢復機制。

- 資源消耗：交易管理需要消耗額外的資源來維護日誌、鎖機制和緩存，可能增加系統的複雜性和運行成本。

- 可擴展性限制：在需要極高可擴展性 (例如：分佈式系統或雲原生應用等) 的系統中，交易的高一致性需求可能會限制系統的擴展能力。

- 複雜的管理和配置：設定和管理交易特性需要一定專業知識，特別是在配置隔離級別和性能優化時，可能會增加系統管理的複雜性。

- 可能影響系統可用性：在某些情況下，為了維持高一致性和隔離性，可能會暫時鎖定某些資源，從而降低系統的可用性，特別是在高負載環境中。
