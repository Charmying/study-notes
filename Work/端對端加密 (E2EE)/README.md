# 端對端加密 (E2EE)

端對端加密 (End-to-end Encryption，簡稱：E2EE) 是一種加密方法，確保訊息在傳輸過程中只有發送端和接收端可以解密和讀取。這一方法可以追溯到 20 世紀 90 年代，Phil Zimmerman 發佈了 Pretty Good Privacy (PGP)。

簡單來說，把 E2EE 想像成一封信，這封信只有寄件人和收件人可以看到內容，郵差等送信人或單位無法閱讀這封信的內容。

<br />

## 未加密訊息如何運作

假設有一個通訊型應用程式，建立一個帳戶 A，想將資訊傳送給帳戶 B，中間必須透過伺服器 (Server) 進行資訊傳遞。

這是一個用戶端-伺服器模型，伺服器充當了發送端和接收端之間的中間方。圖中 A ⮂ S 和 S ⮂ B 之間的訊息是加密的，其中一個例子就是傳輸層安全協議 (Transport Layer Security，簡寫：TLS)，廣泛用於保護用戶端和伺服器之間的連接。

TLS 與 E2EE 一樣使用公開金鑰加密 (公鑰加密) 並確保沒有中間方可以讀取訊息。但是 TLS 是在使用者和伺服器之間實作，而不是在兩個使用者間。這樣只能確保訊息在進出伺服器傳輸過程中的安全，但是伺服器上的訊息仍是處於解密的狀態，而 E2EE 可以解決這個問題。

<br />

## 端對端加密的工作原理

端點儲存用於加密和解密訊息的加密密鑰。此方法採用公鑰加密。

公鑰加密也稱為非對稱加密，使用可以與別人共享的公鑰和私鑰。共享公鑰后，其他人可以使用公鑰來加密訊息并將其發送給公鑰的所有者。只有對應的私鑰，也稱為解密密鑰，才能用於解密訊息。

<br />

## 端對端加密的運作方式

1. 生成密鑰對：每個通訊端都生成一對公鑰和私鑰。公鑰用於加密，私鑰用於解密。

2. 交換公鑰：通訊雙方交換各自的公鑰。通常通過一個安全的通道完成，以防止中間人攻擊 (MITM)。

3. 加密通訊：發送端使用接收端的公鑰加密要傳輸的訊息。由於只有接收端擁有相應的私鑰，所以只有接收端才能解密這些訊息。

4. 解密訊息：接收端使用自己的私鑰解密收到的訊息，從而確保只有發送端和接收端能讀取傳輸內容。

<br />

## 端對端加密的優缺點

### 優點

- 高度隱私：只有通訊的雙方能夠解密訊息，即使是伺服器端也無法讀取通訊內容。

- 訊息保護：防止中間人攻擊，確保傳輸過程中訊息不會被攔截和篡改。

- 安全性增強：使用強大的加密算法，使得解密變得極其困難，保障訊息安全性。

- 信任建立：只有通訊雙方能夠讀取訊息，增強了用戶對於服務的信任。

- 防止訊息洩漏：即使訊息在傳輸過程中被截獲，也無法被解密，因此無法被濫用。

### 缺點

- 難以管理：管理加密密鑰可能是一個問題，尤其是在大型團隊中，可能需要複雜的密鑰管理基礎設施規範。若用戶丟失加密密鑰或密碼，訊息將無法恢復，導致訊息永久丟失。

- 法律和監管問題：E2EE 使得執法機構難以進行合法監控，可能引起法律和監管方面的挑戰。執法部門可能也會更難追查透過 E2EE 加密進行的犯罪活動，因為無法存取這些通訊的內容。

- 複雜的實施：設計和實現安全的 E2EE 需要高水準的技術專業知識，而且容易出現錯誤。

- 性能影響：加密和解密操作會消耗計算資源，可能會影響性能，尤其是在資源有限的設備上。

- 使用限制：可能無法使用某些依賴於數據存取的服務，例如：基於內容的廣告、數據分析等，因為服務提供商無法讀取內容。
