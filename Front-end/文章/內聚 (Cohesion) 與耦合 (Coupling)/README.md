# 內聚 (Cohesion) 與耦合 (Coupling)

在軟體開發中，內聚 (Cohesion) 與耦合 (Coupling) 是衡量程式碼品質的重要指標，影響著程式的可讀性、可維護性與擴展性。這兩個概念通常被一起討論，因為在軟體架構設計中有密切的關聯。

<br />

## 內聚 (Cohesion) - 模組內部的一致性

內聚指的是模組 (Module) 內部的關聯性有多強。也就是說，模組內部的功能是否圍繞同一個目標，或是彼此之間關聯性不大。

內聚度高的模組，其內部各個功能都緊密相關，且通常只專注於一個特定的職責 (Single Responsibility)。這樣的設計能提高可讀性與可維護性，減少後續修改時影響到其他功能的風險。

- 內聚的類型 (由低到高)

    內聚程度從低到高可分為以下幾種：

    1. 偶然內聚 (Coincidental Cohesion)

        - 內部任務幾乎沒有關聯，只是被隨機放在一起。

        - 例子：一個模組內包含「格式化日期」、「計算稅額」和「發送電子郵件」等完全不相關的任務。

    2. 邏輯內聚 (Logical Cohesion)

        - 內部任務執行類似的操作，但實際任務不同，通常由條件判斷決定要執行哪個任務。

        - 例子：同一個函式負責「讀取檔案」與「讀取資料庫」，但根據參數決定具體執行哪一個。

    3. 時間內聚 (Temporal Cohesion)

        - 內部任務會在「相同時間點」被執行，彼此之間沒有直接關聯。

        - 例子：`init()` 初始化函式負責「開啟日誌」、「連接資料庫」、「載入設定檔」等。

    4. 程序內聚 (Procedural Cohesion)

        - 內部任務按照某種程序順序執行，但彼此之間的關聯性較弱。

        - 例子：`processOrder()` 先檢查「庫存」，再「計算運費」，最後「產生發票」，但每個步驟彼此相對獨立。

    5. 通訊內聚 (Communicational Cohesion)

        - 內部任務使用相同的資料，並在處理同一組資料時被執行。

        - 例子：`calculateSalary()` 同時「計算基本薪資」、「扣稅」與「津貼」，因為這些任務都依賴員工的薪資資訊。

    6. 任務內聚 (Functional Cohesion)

        - 內部任務共同達成單一的目標，每個任務都是必要且密切相關的。

        - 例子：`encryptPassword()` 只負責「加密密碼」，而不會順便進行其他不相關的處理。

## 耦合 (Coupling) - 模組之間的依賴程度

耦合指的是模組與模組之間的依賴程度。當模組之間的關聯性過強時，修改一個模組可能會影響到其他模組，導致系統難以維護與擴展。

高內聚通常伴隨低耦合，而低耦合能提升系統的靈活性，使得各個部分能夠獨立變更或替換，減少影響範圍。

- 耦合的類型 (由高到低)

    耦合程度從高到低可分為以下幾種：

    1. 內容耦合 (Content Coupling)

        - 一個模組直接修改或存取另一個模組的內部數據。

        - 缺點：任何修改內部數據的結構，都會影響其他模組。

        - 例子：ModuleA 直接存取 ModuleB 的變數 b.value = 10;。

    2. 公用耦合 (Common Coupling)

        - 多個模組共享同一組全域變數。

        - 缺點：任何模組修改了全域變數，其他模組可能會受到影響。

        - 例子：`config.js` 定義了一個 `globalConfig`，所有模組都直接使用。

    3. 外部耦合 (External Coupling)

        - 兩個模組依賴於相同的外部工具或框架。

        - 缺點：若外部工具變更，所有模組都需調整。

        - 例子：所有模組都強依賴某個特定的資料庫，例如：MySQL 若要改用 PostgreSQL，則所有模組都需修改。

    4. 控制耦合 (Control Coupling)

        - 一個模組透過參數告知另一個模組該執行哪種流程。

        - 缺點：增加了模組間的關聯性，使得修改變得困難。

        - 例子：`processData(mode: string)`，當 `mode="json"` 時執行 JSON 解析，當 `mode="xml"` 時執行 XML 解析。

    5. 標記耦合 (Stamp Coupling)

        - 模組之間透過複雜的資料結構傳遞資訊，而不是單純的基本型別 (例如：數字 or 字串)。

        - 缺點：若資料結構變更，所有使用這個的模組都會受到影響。

        - 例子：`processUser(user: User)`，若 `User` 結構改變，所有依賴 `User` 的函式都要修改。

    6. 資料耦合 (Data Coupling)

        - 模組之間只透過簡單的資料型別 (例如：數字 or 字串) 傳遞資訊，而不依賴整個結構。

        - 優點：低影響性，模組可獨立運行。

        - 例子：`calculateTotal(price: number, tax: number)` 只接收基本數據，與其他模組的關聯性低。

## 如何提升內聚、降低耦合

-  提升內聚的方法

    - 單一職責原則 (Single Responsibility Principle，簡稱：SRP)

        - 每個模組應該只專注於一項功能，避免額外的責任。

    - 封裝 (Encapsulation)

        - 隱藏模組內部的細節，僅暴露必要的功能。

    - 命名清晰，結構合理

        - 讓每個函式與模組的名稱與任務相符，避免混雜不同類型的任務。

- 降低耦合的方法

    - 使用介面與依賴注入 (Dependency Injection，簡稱：DI)

        - 透過介面降低模組之間的直接依賴，例如：Angular 和 Spring 都使用 DI 來減少耦合。

    - 避免全域變數

        - 盡量減少共享狀態，改用函式參數或回傳值傳遞資料。

    - 使用事件驅動架構

        - 讓模組透過事件溝通，而非直接呼叫彼此的方法，例如：Vue 的事件機制或 `RxJS`。

## 總結

「高內聚、低耦合」是良好軟體設計的目標。提高內聚能讓模組更專注且易於維護，而降低耦合則讓系統更靈活，便於擴展。掌握這兩大概念，能讓程式碼更具彈性與可讀性，提升開發效率與品質。
