# CSR 與 SSR

在網頁開發中，CSR (Client-Side Rendering，用戶端渲染) 和 SSR (Server-Side Rendering，伺服器端渲染) 是兩種常見的渲染方式。這兩者決定了網頁內容是如何生成並呈現的。

<br />

## CSR (Client-Side Rendering，用戶端渲染)

CSR 是指網頁的內容並不是由伺服器預先生成，而是交由使用者的瀏覽器通過 JavaScript 動態產生。簡單來說，當點開一個使用 CSR 的網頁時，伺服器只會回傳一個非常簡單的 HTML 文件，通常這個文件裡只有一個空的容器 (例如：`<div id="app"></div>`) 和一些 JavaScript 檔案。接著瀏覽器會下載並執行這些 JavaScript，通過程式碼向伺服器請求資料 (例如：透過 API)，然後將資料轉換成看到的網頁內容。

Vue、React、Angular 等框架預設就是 CSR。

### CSR 的運作流程

1. 點擊網頁連結，瀏覽器向伺服器發出請求。

2. 伺服器回傳一個基本的 HTML 文件，裡面包含 JavaScript 檔案的連結。

3. 瀏覽器下載並執行 JavaScript。

4. JavaScript 向伺服器拿取所需的資料。

5. 瀏覽器根據這些資料動態生成網頁內容，讓使用者看到完整的頁面。

### CSR 的優點

- 流暢的使用者體驗：因為內容是動態生成的，在瀏覽網頁時可以順暢的交互，例如：點擊按鈕後無需重新載入整個頁面就能更新內容。

- 減輕伺服器負擔：伺服器只需要提供基本的 HTML 和資料，真正的渲染工作由瀏覽器完成，這樣伺服器就不用承擔太多壓力。

- 適合現代前端框架：CSR 常搭配像是 React 或 Vue 前端框架，讓開發人員能更輕鬆打造複雜的用戶界面。

### CSR 的缺點

- 初始載入速度慢：因為瀏覽器需要先載入 JavaScript 並執行渲染，可能會在網頁完全顯示前看到空白頁面或「載入中」的提示。

- 不利於 SEO：搜尋引擎 (例如：Chrome、Safari) 在爬取網頁時，可能無法順利執行 JavaScript，導致只能看到基本的 HTML，無法抓到動態生成的內容，這樣就會影響網頁的搜尋排名。

- 依賴 JavaScript：若瀏覽器關閉了 JavaScript，網頁就沒辦法正常顯示。

<br />

## SSR (Server-Side Rendering，伺服器端渲染)

SSR 是指網頁的內容在伺服器上預先生成，然後直接將完整的 HTML 文件傳送到瀏覽器。當訪問一個使用 SSR 的網頁時，伺服器會根據請求生成一個包含所有內容的 HTML 文件，瀏覽器收到後馬上就能顯示出來，不需要額外執行 JavaScript 來渲染。

React 可以透過 Next.js、Vue 可以透過 Nuxt.js、Angular 可以透過 官方的 Angular Universal 實現 SSR。

### SSR 的運作流程

1. 點擊網頁連結，瀏覽器向伺服器發出請求。

2. 伺服器根據請求生成完整的 HTML 文件。

3. 伺服器把這個 HTML 文件傳送到瀏覽器。

4. 瀏覽器直接顯示內容，立刻就能看到完整的網頁。

### SSR 的優點

- 快速的初始載入速度：因為瀏覽器收到的是已經渲染好的 HTML，不用等 JavaScript 執行，可以更快看到網頁內容。

- 利於 SEO：搜尋引擎能直接抓到完整的 HTML 內容，這對提高網頁的搜尋排名很有幫助。

- 不依賴 JavaScript：即使瀏覽器禁用了 JavaScript，網頁還是能顯示基本的內容。

### SSR 的缺點

- 伺服器負擔較大：每次訪問網頁，伺服器都要動態生成 HTML，尤其在流量大的時候，可能會讓伺服器吃不消。

- 使用者體驗稍弱：SSR 在切換頁面時可能需要重新載入整個頁面，無法像 CSR 那樣提供無縫的交互體驗。

- 開發較複雜：SSR 需要在伺服器端處理渲染，對於開發人員和維護來說可能會更麻煩一些。

<br />

## CSR 和 SSR 的比較

|  | CSR (用戶端渲染) | SSR (伺服器端渲染) |
| - | - | - |
| 渲染地點 | 瀏覽器端 | 伺服器端 ||
| 初始載入速度 | 較慢 (需載入並執行 JavaScript) | 較快 (直接收到完整 HTML) |
| SEO 效果 | 不佳 (需額外優化) | 優秀 (搜尋引擎可直接抓取內容) |
| 用戶體驗 | 流暢 (適合互動性高的應用) | 普通 (頁面切換可能需重新載入) |
| 伺服器負擔 | 較輕 (渲染交給瀏覽器) | 較重 (伺服器需處理渲染) |

<br />

## CSR 和 SSR 的應用場景

### 選擇 CSR 的場景

- 需要高度互動性：單頁應用 (SPA)、社交媒體平台或線上編輯工具，這些都需要流暢的使用者體驗，CSR 是更好的選擇。

- SEO 不是重點：後台管理系統或公司內部工具，這類網頁不需要在意搜尋引擎排名。

- 減輕伺服器壓力：若伺服器資源有限，CSR 可以把渲染工作交給用戶的瀏覽器。

### 選擇 SSR 的場景

- 重視 SEO：新聞網站、部落格或電商平台，這些都需要讓搜尋引擎順利抓取內容。

- 追求快速載入：行銷用的 landing page，快速顯示內容能提高使用者留存率。

- 內容較靜態：若網頁不需要太多動態交互，SSR 能提供簡單又高效的解決方案。

<br />

## 總結

CSR 和 SSR 各有自己的優勢和局限。CSR 適合需要高度互動和流暢體驗的網頁，但初始載入速度和 SEO 是弱點。SSR 則在載入速度和 SEO 上表現出色，但可能會增加伺服器負擔並影響交互體驗。在實際開發中，沒有一種方式是絕對完美的，選擇哪一種取決於專案目標。若網頁需要兼顧兩者的優點，也可以考慮混合渲染的方式，例如：初次載入用 SSR，後續交互用 CSR，這樣就能打造出既快速又好用的網頁體驗。
